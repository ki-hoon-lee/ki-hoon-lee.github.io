Overview

An Autonomous Robotic Goalkeeper

GoalE is a real-time autonomous ball tracking and interception system — a robotic goalkeeper that uses four ultrasonic sensors to triangulate a ball's position, predict its trajectory, and actuate a servo-driven goalie to intercept it before it crosses the goal line. Built by a team of four as a final project for UC Berkeley's ME235 Mechatronics course.

The motivation was simple: build something genuinely fun that pushed the limits of embedded systems design — real-time multi-tasking, sensor fusion, trajectory prediction, and a live LabVIEW GUI all running in parallel on an ESP32 microcontroller.

System Architecture

Hardware + Software Stack

HARDWARE
ESP32 + Sensor Array

An ESP32 microcontroller drives four ultrasonic sensors mounted at the corners of a 56×86cm playing field. A servo motor actuates the goalie arm, and an IR beam sensor at the goal line detects when a goal is scored. All sensors run concurrently via FreeRTOS task scheduling.

SOFTWARE
C / FreeRTOS + LabVIEW GUI

The ESP32 firmware is written in C using FreeRTOS for real-time multi-task execution. Position data is streamed over UART to a LabVIEW GUI which live-plots ball position, displays goalie location, tracks goal count, and shows ball speed in real time.

LOCALISATION
Least Squares Triangulation

With distance readings from all four sensors, ball position is estimated using a least squares multilateration algorithm — forming a linear system from the distance equations, solving via the normal equations (AᵀA)⁻¹Aᵀb to yield an (x, y) coordinate each cycle.

TRAJECTORY
Flip-Flop Buffer Prediction

A custom double-buffer (flip-flop) architecture allows position data to be collected continuously while the trajectory task reads the previous buffer in parallel. The ball's predicted goal-line crossing point is calculated from two averaged half-trajectory vectors.

Real-Time Multitasking

Four Parallel FreeRTOS Tasks

The system runs four concurrent tasks with carefully tuned priorities to ensure data gathering is never starved by computation — a key design challenge given the tight sensor timing requirements.

PRIORITY 7 — HIGHEST
Ultrasonic Task

Fires all four sensors in sequence with inter-sensor delays to prevent interference. Computes least-squares position and writes to the active position buffer. Runs every ~41ms.

PRIORITY 6
IR Goal Sensor Task

Monitors the infrared beam at the goal line. Detects beam breaks to register goals, sets the goal flag, and debounces the signal to avoid duplicate counts.

PRIORITY 5
Trajectory Task

Swaps the flip-flop buffer, reads the filled position buffer, calculates the ball's predicted x-intercept at the goal line, maps it to a servo angle and actuates the goalie.

PRIORITY 5
UART Send Task

Serialises current ball position, goalie position and goal status into a compact string and streams it over UART to the LabVIEW GUI at 20ms intervals.

Engineering Highlights

Notable Implementation Details

Flip-flop buffer logic — the most elegant part of the codebase. Rather than locking the position buffer during trajectory calculation (which would stall data collection), two buffers are used alternately. When the trajectory task runs, it atomically swaps which buffer is being written to, then reads the now-frozen previous buffer in parallel with ongoing data collection:

if(flip) {
    positionBuffer = &positionsFlop;
    samples = positionsFlip;
} else {
    positionBuffer = &positionsFlip;
    samples = positionsFlop;
}
flip = !flip;
positionBufferIndex = 0;
float xIntercept = calculate_trajectory(samples, sampleCount);
Flip-flop buffer swap — trajectory task reads stale buffer while new data fills the other

Empirical motor angle mapping — trigonometric calculation of the required servo angle proved inaccurate in practice due to mechanical non-linearities. Instead, a quadratic polynomial was fitted to empirical calibration data, mapping x-intercept position directly to the servo pulse width:

float get_motor_angle(int xIntercept) {
    float rotation_angle;
    rotation_angle = (0.1613 * xIntercept * xIntercept)
                   - (17.77 * xIntercept)
                   + 447.75;
    return rotation_angle;
}
Empirically calibrated quadratic mapping from goal-line intercept (cm) to servo angle (°)
LabVIEW GUI

Real-Time Monitoring Interface

The LabVIEW GUI provides a complete real-time view of the system state. It receives the UART data stream from the ESP32 and renders it live, featuring:

Live ball tracking — an XY plot displays a moving circle representing the ball's current position within the 56×86cm field, or can be toggled to show a position-over-time graph.

Goalie indicator — a visual bar shows the current goalie position along the goal line in real time.

Goal counter — an LED indicator flashes on goal detection and a running total is maintained, resettable via a dedicated button.

Speed display — ball speed is computed from successive position readings and displayed numerically. Start/Stop controls prevent data backlogging between sessions.
